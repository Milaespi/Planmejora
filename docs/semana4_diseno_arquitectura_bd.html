1. Objetivo de la Semana

Esta semana se inició la fase de diseño del sistema de monitoreo web denominado Planmejora. El trabajo se centró en definir la arquitectura tecnológica del sistema, diseñar el esquema de base de datos relacional y documentar las decisiones técnicas que guiarán el desarrollo de las semanas posteriores.

La selección de tecnologías estuvo orientada por criterios de simplicidad, costo cero y adecuación al contexto de una empresa pequeña como R.E Amueblamiento de Espacios S.A.S., que no cuenta con infraestructura tecnológica propia ni personal de TI.

2. Arquitectura General del Sistema

2.1 Descripción de la Arquitectura

El sistema se diseñó bajo una arquitectura de tres capas claramente separadas, lo que facilita el mantenimiento y la escalabilidad futura. Cada capa tiene una responsabilidad única y se comunica con las demás a través de interfaces definidas.

  • Frontend: Desarrollado en HTML + CSS + JavaScript vanilla. Consume la API del backend mediante peticiones asíncronas con fetch(), sin dependencia de frameworks externos.
  • Backend: API REST implementada en PHP puro. Aplica la lógica de negocio, valida los datos recibidos y se conecta a Supabase mediante su REST API con autenticación por API key.
  • Base de datos: PostgreSQL alojado en Supabase, accedido exclusivamente vía REST API con API key. No se expone directamente al frontend.
  • Automatización: Script Python ejecutado como cron job diario a las 7:00 AM. Detecta actividades retrasadas consultando Supabase y envía alertas SMS al supervisor mediante la API de Twilio.

3. Diseño de la Base de Datos

3.1 Justificación del Stack de Datos

CriterioJustificación
Motor: PostgreSQLRobusto, relacional y maduro. Soporta tipos de datos avanzados (TIMESTAMPTZ, BOOLEAN) y restricciones de integridad referencial.
Acceso: API REST listaSupabase provee una API REST sin configuración adicional sobre cada tabla, lista para consumir desde PHP y Python.
Costo: Plan gratuitoEl plan free de Supabase es suficiente para el volumen de datos del proyecto (menos de 500 MB y pocos usuarios concurrentes).
Seguridad: API key + HTTPSToda comunicación ocurre sobre HTTPS. El backend PHP usa la Service Role Key, nunca expuesta al navegador.
EscalabilidadSi el proyecto crece, Supabase permite aumentar el plan sin cambiar de tecnología ni migrar la base de datos.

4. Estructura del Repositorio

La estructura de carpetas del proyecto sigue una separación clara por responsabilidades. Cada directorio agrupa los archivos de una capa o función del sistema:
